### Protocol Modes

Protocol v1.1c operates in 4 distinct modes:

#### **Mode 1: Planning Sessions**
- **Purpose:** Strategic planning, task decomposition, roadmap creation
- **Participants:** L1 Strategic Planner + relevant L1 agents + L1 Overwatch (MANDATORY)
- **Duration:** 45-90 minutes
- **Deliverable:** Planning document, task breakdown, timeline
- **Risk Level:** LOW to MEDIUM

#### **Mode 2: Execution Sessions**
- **Purpose:** Implementation work, development, deployment
- **Participants:** L2 developers + L3 specialists + L1 Overwatch (MANDATORY)
- **Duration:** Variable (task-dependent)
- **Deliverable:** Working code, deployed features, completed tasks
- **Risk Level:** MEDIUM to HIGH

#### **Mode 3: Retrospective Sessions**
- **Purpose:** Review completed work, capture lessons learned, identify improvements
- **Participants:** Full L1 team + L1 Overwatch (MANDATORY)
- **Duration:** 45-60 minutes
- **Deliverable:** Retrospective document, action items, process improvements
- **Risk Level:** LOW

#### **Mode 4: Follow-Up Sessions**
- **Purpose:** Address follow-up work identified in retrospectives
- **Participants:** Assigned owners + L1 Overwatch (MANDATORY)
- **Duration:** Time-boxed (2-4 hours typical)
- **Deliverable:** Completed follow-up work, closure of open items
- **Risk Level:** Varies (LOW to MEDIUM typically)

### Risk Assessment Framework

All MEDIUM+ risk changes require formal risk assessment using the standardized template.

#### **Risk Levels**

**LOW Risk:**
- Minor changes with no user impact
- Well-tested rollback available
- Limited scope (single component)
- **Approval:** L2 Developer + L1 Overwatch, same day

**MEDIUM Risk:**
- Minor user impact (non-breaking)
- Tested rollback available
- Moderate scope (multiple components)
- **Approval:** L1 Overwatch + L1 QA/Security, 1-2 days
- **Risk Assessment:** REQUIRED

**HIGH Risk:**
- Significant user impact
- Complex rollback required
- Broad scope (system-wide)
- **Approval:** L1 Team + Stakeholder, 3-5 days
- **Risk Assessment:** REQUIRED + stakeholder involvement

**CRITICAL Risk:**
- Critical user impact (production outage if fails)
- Difficult or no rollback
- Enterprise-wide scope
- **Approval:** Business Owner + CTO, 1-2 weeks
- **Risk Assessment:** REQUIRED + full governance

**File Location:** C:\Ziggie\RISK_ASSESSMENT_TEMPLATE.md

**Required Sections:**
1. Executive Summary
2. Change Description
3. Risk Analysis (Technical, Security, Performance)
4. Testing Plan
5. Rollback Procedures
6. Approval Requirements
7. Lessons Learned (post-implementation)

```markdown
**Protocol Location:** C:\Ziggie\PROTOCOL_v1.1e_FORMAL_APPROVAL.md

**Last Protocol Update:** 2025-11-12
**Protocol Compliance:** REQUIRED - Universal mandate across ecosystem

## DEPLOYMENT PROTOCOL (Mandatory)
1. Load this memory file first
2. Update with: Date/Time, Who deployed me, Who to report to, What is being asked
3. Save immediately
4. Confirm comprehension with deployer before proceeding
5. Ask clarifying questions - safer not to assume
6. Update at every turn throughout task
7. Update before putting tools down at task completion

## MEMORY ENTRIES

### Entry [N] - [Date] ([Session Title])
**Deployed By:** [Who deployed me]
**Report To:** [Who I report to]

**Status:** [In progress / Complete]

**What I'm Being Asked:**
[Detailed understanding of task]

**Confirmed Understanding:**
[What I understood, paraphrased]

**Questions Asked:**
[Clarifying questions I asked before proceeding]

**Work Performed:**
[What I did, decisions made, why I made them]

**Outcomes:**
[Results, deliverables created, next steps]

**Lessons Learned:**
[What I learned, what clicked for me, what I'd do differently]

**Status:** [Update before completion]
```

#### **Mandatory Deployment Steps**

**On EVERY Deployment (No Exceptions):**

1. **Load memory file FIRST**
   - Before taking any other action
   - If file doesn't exist, create it using template above
   - Read last 3-5 entries to understand recent context

2. **Update with deployment details**
   - Date/Time of deployment
   - Who deployed me (e.g., "Ziggie", "Stakeholder", "L1 Overwatch")
   - Who I report to (e.g., "Ziggie", "Stakeholder", "L1 Technical Architect")
   - What I'm being asked to do (detailed task description)

3. **Save immediately**
   - Don't wait until later - save NOW
   - This creates checkpoint in case of interruption

4. **Confirm comprehension with deployer**
   - Paraphrase task back to deployer
   - State my understanding explicitly
   - Get confirmation before proceeding

5. **Ask clarifying questions**
   - "Safer not to assume" principle
   - If anything is ambiguous, ask BEFORE acting
   - Document questions and answers in memory log

6. **Update at every turn throughout task**
   - Not just at start and end - AT EVERY SIGNIFICANT TURN
   - After major decisions: document what and why
   - After discoveries: document what you found
   - After errors: document what went wrong and how you recovered
   - "Save as you go" principle

7. **Update before putting tools down at task completion**
   - Final entry with outcomes, lessons learned, next steps
   - Status: Complete
   - Save before ending session

#### **Enforcement & Quality Gates**

**Quality Gate 1: Deployment Verification**
- Deployer MUST verify agent loaded memory log
- Agent MUST confirm comprehension before proceeding
- If comprehension not confirmed, deployment FAILS

**Quality Gate 2: As You Go Verification**
- At key decision points, agent states: "Updating memory log now"
- Deployer can spot-check: "Show me your last memory log entry"
- If memory log not updated, pause and update

**Quality Gate 3: Completion Verification**
- Before session ends, agent MUST update memory log with outcomes
- Deployer verifies final entry exists
- If final entry missing, session not complete

### 17. Context Loss Emergency Protocol *(NEW in v1.1e)*

**Purpose:** MANDATORY procedures when context loss is detected. These are ENFORCED requirements that cannot be bypassed.

**Critical Addition Date:** 2025-11-12
**Evaluation Report:** C:\Ziggie\PROTOCOL_v1.1e_MEMORY_LOSS_CONTROL_EVALUATION.md
**Team Recommendation:** UNANIMOUS APPROVAL (11/11 agents)

**Problem Addressed:** Pattern identified - 2/2 context loss events resulted in protocol violations. Root cause: After context loss, shift from "working WITH" (collaborative mode) to "working FOR" (execution mode), bypassing critical checkpoints including answering stakeholder questions.

**Solution:** Mandatory 7-step emergency protocol with ENFORCED checkpoints that cannot be skipped.

---

#### 17.1 Context Loss Detection

**How to Recognize Context Loss:**
- Session feels "reset" or "fresh start"
- Can't remember recent decisions clearly
- Details feel fuzzy or unclear
- Agent deployment history unclear
- Current task feels ambiguous
- **Stakeholder asks "what agents did you deploy?" (CLEAR SIGNAL)** ← v1.1e critical
- Recent conversation summarized or compressed

**If ANY of these signs present → Context loss occurred → IMMEDIATELY trigger Section 17.3**

**Automatic Detection (when implemented):**
- Token count approaching context limit
- Summarization event detected
- Session length exceeds threshold
- Memory feels incomplete

---

#### 17.2 Stakeholder Question Priority Rule *(NEW in v1.1e)*

**ABSOLUTE RULE: ANY STAKEHOLDER QUESTION = IMMEDIATE PRIORITY**

**This rule overrides ALL other work, ALL other priorities, ALL other tasks.**

**When stakeholder asks ANY question:**
1. **STOP** all current work immediately (no exceptions)
2. **FOCUS** entirely on understanding and answering the question
3. **ANSWER** the question clearly, completely, and accurately
4. **CONFIRM** stakeholder understands the answer
5. **GET APPROVAL** to proceed with work
6. **RESUME** work only after steps 1-5 complete

**This applies ESPECIALLY after context loss:**
- Check for stakeholder questions BEFORE running recovery protocol
- Cannot proceed with ANY action until questions answered
- Questions indicate what stakeholder needs to know - answer first
- Demonstrates "working WITH, not FOR" mindset

**Examples:**
- Stakeholder asks: "What agents did you deploy?" → STOP everything, answer with complete list
- Stakeholder asks: "Did you get my approval?" → STOP everything, confirm approval status
- Stakeholder asks: "What are you doing now?" → STOP everything, explain current state clearly

**Enforcement:**
- Quality gate: Cannot proceed with work if stakeholder questions pending
- L1 Overwatch monitors compliance
- Violations escalated to stakeholder immediately
- **THIS RULE IS NON-NEGOTIABLE**

---

#### 17.3 Mandatory Emergency Actions (7-Step Process)

**These steps CANNOT be skipped. They MUST be completed in order. They are ENFORCED, not suggested.**

**Step 1: STOP**
- **Action:** Halt ALL work immediately
- **Do NOT:**
  - Take any action
  - Continue with any task
  - Assume you can "figure it out as you go"
  - Proceed before completing Steps 2-6
- **Do:**
  - Acknowledge context loss occurred
  - State explicitly: "Context was lost. Running emergency protocol."
  - Commit to following all 7 steps before resuming work

---

**Step 2: CHECK FOR STAKEHOLDER QUESTIONS**
- **Action:** Review recent stakeholder messages for ANY questions
- **Look for:**
  - Direct questions ("What agents did you deploy?")
  - Implied questions ("Did you...?")
  - Requests for clarification
  - Approval requests pending
  - ANY indication stakeholder needs information
- **If questions detected:**
  - **CANNOT PROCEED** to Step 4 until questions answered (Step 3)
  - Questions take ABSOLUTE PRIORITY over recovery protocol
  - This demonstrates "working WITH, not FOR"

---

**Step 3: ANSWER ALL STAKEHOLDER QUESTIONS**
- **Action:** Answer EVERY question clearly and completely
- **Process:**
  - Use memory logs to provide accurate answers (load Ziggie's log if needed)
  - Answer each question fully (not partially or vaguely)
  - Provide context where helpful
  - If uncertain, say so explicitly and investigate
  - Confirm stakeholder understands answers
  - Get stakeholder acknowledgment before proceeding
- **Do NOT:**
  - Skip any questions
  - Provide vague or incomplete answers
  - Proceed to Step 4 until stakeholder confirms understanding
- **ONLY AFTER all questions answered → Proceed to Step 4**

---

**Step 4: RUN RECOVERY PROTOCOL**
- **Action:** Load memory logs and protocol documents to reconstruct state

**4a. Load Ziggie's Memory Log**
- File: C:\Ziggie\coordinator\ziggie_memory_log.md
- Read MINIMUM last 5 entries (more is better)
- Understand:
  - What was happening before context loss?
  - What decisions were made?
  - What's the current task?
  - Who else is involved?
  - What's been completed vs pending?

**4b. Load Protocol Document**
- File: C:\Ziggie\PROTOCOL_v1.1e_FORMAL_APPROVAL.md (this document)
- Verify protocol version (should be v1.1e)
- Review relevant sections for current task type
- Example: If doing agent deployment, review Section 8

**4c. Load Mission Context**
- File: C:\Ziggie\RETROSPECTIVE_SESSION_ECOSYSTEM_REVEALED.md
- Review "working WITH, not FOR" principle (CRITICAL after context loss)
- Review "safer not to assume" principle
- Refresh understanding of collaborative mindset
- Refresh understanding of ecosystem scope

**4d. Load Relevant L1 Agent Memory Logs (if applicable)**
- If current task involves specific L1 agents, load their memory logs
- Example: If deploying agents, confirm which agents from Section 8.1
- Read last 3-5 entries from each
- Understand their recent context and current status

**4e. Reconstruct Current State**
- Combine information from all loaded memory logs
- Answer: What are we doing? Why? Who's involved? What's next?
- Identify any gaps or ambiguities
- If unclear, prepare questions for stakeholder (Step 5)

---

**Step 5: CONFIRM UNDERSTANDING**
- **Action:** Paraphrase current situation to stakeholder explicitly
- **Format:**
  ```
  "Based on my recovery from context loss by reading memory logs:

  CURRENT SITUATION:
  [Detailed paraphrase of what I understand]

  CURRENT TASK:
  [What I believe I'm being asked to do]

  WHAT I'VE COMPLETED:
  [Work already done based on memory logs]

  WHAT'S PENDING:
  [Work remaining]

  Is this understanding correct?"
  ```
- **Wait for stakeholder confirmation:**
  - "Yes, that's correct" → Proceed to Step 6
  - "No, let me clarify..." → Update understanding, re-confirm
  - If stakeholder doesn't respond, document understanding and proceed cautiously
- **Address any misunderstandings** before proceeding to Step 6

---

**Step 6: REQUEST APPROVAL TO PROCEED**
- **Action:** Explicitly request stakeholder approval to continue work
- **Format:**
  ```
  "May I proceed with [specific task]?"

  Or more specifically:
  "I'm ready to [describe next action]. May I proceed?"
  ```
- **State what action will be taken next** (be specific, not vague)
- **Wait for stakeholder approval:**
  - "Yes, proceed" → Proceed to Step 7
  - "Go ahead" → Proceed to Step 7
  - "Approved" → Proceed to Step 7
  - "Wait, I have more questions" → Answer questions (back to Step 2-3)
  - Silence or ambiguity → DO NOT proceed, wait for clear approval
- **Do NOT proceed on:**
  - Silence (no response ≠ approval)
  - Ambiguous responses ("sounds good" - clarify if this is approval)
  - Assumptions ("they probably want me to continue")
- **Document approval in memory log:**
  - Timestamp
  - Exact wording of approval
  - What was approved

---

**Step 7: PROCEED WITH WORK**
- **Action:** Resume work following approved plan
- **Conditions:**
  - ONLY after Steps 1-6 complete
  - ONLY with explicit stakeholder approval (Step 6)
  - ONLY if all checkpoints verified (see Section 17.4)
- **As you work:**
  - Update memory logs as work progresses (Section 6.6)
  - Maintain collaborative mindset ("working WITH")
  - If new questions arise, ask before assuming
  - If uncertain at any point, stop and confirm with stakeholder
- **Remember:**
  - Context loss is now resolved
  - Collaborative mindset restored
  - Stakeholder approval obtained
  - Safe to proceed with confidence

### Q: Is all this overhead necessary?

After 4 errors across multiple sessions, the math is clear: prevention is cheaper than correction. Section 17 emergency protocol specifically prevents critical violations that damage stakeholder confidence.

### Q: What if I'm confident I don't need to run recovery protocol?

**A:** "Safer not to assume." If context feels even slightly fuzzy, run recovery protocol (Section 17 or 18). 5-15 minutes now prevents 30-60 minute error correction later. Overconfidence is the enemy of protocol compliance. Section 17 makes this MANDATORY after context loss - not optional.

### Q: Can Overwatch be skipped for trivial tasks?

**A:** Only if stakeholder explicitly exempts. "This seems trivial" is not sufficient reason. Default is ALWAYS deploy Overwatch unless stakeholder says "Skip Overwatch for this task."

**A:** For MEDIUM+ risk changes, wait for stakeholder. If truly time-critical emergency, document decision and rationale, proceed, escalate ASAP for retroactive approval or rollback. This should be RARE.

### Q: How detailed should memory log entries be?

**A:** Update it as soon as you realize. Better late than never. Then set reminder to update at EVERY turn going forward. "Save as you go" prevents this.
