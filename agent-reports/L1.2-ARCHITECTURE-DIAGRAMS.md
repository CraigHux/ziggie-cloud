## 6. PROCESS LIFECYCLE DIAGRAM

```
┌─────────────────────────────────────────────────────────────────┐
│                    AGENT LIFECYCLE                              │
└─────────────────────────────────────────────────────────────────┘

┌──────────┐
│  PENDING │  Request received, not yet processed
└──────────┘
      │
      │ Coordinator picks up request
      ▼
┌──────────┐
│SPAWNING  │  Creating process, agent definition
└──────────┘
      │
      │ subprocess.Popen() successful
      ▼
┌──────────┐
│ DEPLOYED │  Process started, PID assigned
└──────────┘
      │
      │ Agent begins execution
      ▼
┌──────────┐
│ RUNNING  │  ◄──── Parent can check status here
└──────────┘       Parent can read logs
      │            Parent can send commands
      │            Agent can deploy sub-agents
      │
      ├───────────────────┐
      │                   │
      │ Task complete     │ Parent terminates
      │                   │ or timeout
      ▼                   ▼
┌──────────┐        ┌──────────┐
│COMPLETED │        │TERMINATED│
└──────────┘        └──────────┘
      │                   │
      └───────┬───────────┘
              │
              │ Process cleanup
              ▼
┌──────────────────┐
│    ARCHIVED      │  Logs preserved, stats recorded
└──────────────────┘

State Transitions:
PENDING → SPAWNING → DEPLOYED → RUNNING → COMPLETED
                                    ↓
                               TERMINATED

Error States:
PENDING → SPAWNING → FAILED (if spawn fails)
RUNNING → FAILED (if error during execution)

Monitoring Points:
- DEPLOYED: Check PID exists
- RUNNING: CPU/memory usage, log output
- COMPLETED: Exit code, final logs
- TERMINATED: Cleanup successful
```

## 9. COMPARISON: FILE VS API

```
┌─────────────────────────────────────────────────────────────────┐
│                    FILE-BASED                                   │
└─────────────────────────────────────────────────────────────────┘

Advantages:
✅ Simple implementation (4-8 hours)
✅ No service dependencies
✅ Easy debugging (just read files)
✅ Transparent state (visible in file system)
✅ Works immediately on any system

Disadvantages:
⚠️ File I/O overhead (~100-500ms)
⚠️ Manual cleanup needed
⚠️ Not "real-time"
⚠️ Limited scalability (hundreds of agents)

Best for:
- MVP and initial testing
- Simple deployments
- Development environment
- Learning the system

┌─────────────────────────────────────────────────────────────────┐
│                       REST API                                  │
└─────────────────────────────────────────────────────────────────┘

Advantages:
✅ Fast (~50-100ms latency)
✅ Real-time updates (WebSocket)
✅ Scalable (thousands of agents)
✅ Rich monitoring (CPU, memory, logs)
✅ Multiple clients supported
✅ Standard HTTP tooling

Disadvantages:
⚠️ Requires persistent service
⚠️ More complex (12-16 hours)
⚠️ Network dependencies
⚠️ Harder to debug (need logs/monitoring)

Best for:
- Production deployments
- Large-scale operations
- Dashboard integration
- Multiple parent agents
```
