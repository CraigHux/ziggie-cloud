**Protocol v1.1e Section 16: Context Loss Emergency Protocol**

**16.1 Purpose**
This section provides MANDATORY procedures when context loss is detected. These are not suggestions - they are enforceable requirements that cannot be bypassed.

**16.2 Context Loss Detection**

**How to Recognize Context Loss:**
- Session feels "reset" or "fresh start"
- Can't remember recent decisions clearly
- Details feel fuzzy or unclear
- Agent deployment history unclear
- Current task feels ambiguous
- Stakeholder asks "what agents did you deploy?" (clear signal)

**If ANY of these signs present → Context loss occurred → Trigger emergency protocol**

**16.3 Mandatory Emergency Actions (ENFORCED)**

**These steps CANNOT be skipped. They MUST be completed in order.**

**Step 1: STOP**
- Halt ALL work immediately
- Do NOT take any action
- Do NOT continue with any task
- Do NOT assume you can "figure it out as you go"

**Step 2: CHECK FOR STAKEHOLDER QUESTIONS**
- Review recent stakeholder messages
- Identify ANY questions (direct or implied)
- If questions detected → CANNOT PROCEED until answered
- Questions take ABSOLUTE PRIORITY over all work

**Step 3: ANSWER ALL STAKEHOLDER QUESTIONS**
- Answer EVERY question clearly and completely
- Use memory logs to provide accurate answers
- Confirm stakeholder understands answers
- Get stakeholder acknowledgment
- ONLY THEN proceed to recovery

**Step 4: RUN RECOVERY PROTOCOL**
- Load Ziggie's memory log (last 5 entries minimum)

- Load relevant L1 agent memory logs (if applicable)

- Paraphrase current situation to stakeholder

- Address any misunderstandings

- Wait for stakeholder approval ("Yes, proceed" / "Go ahead" / "Approved")

- Document approval in memory log

**Step 7: PROCEED WITH WORK**
- ONLY after Steps 1-6 complete
- Resume work following approved plan
- Update memory logs as work progresses
- Maintain collaborative mindset ("working WITH")
